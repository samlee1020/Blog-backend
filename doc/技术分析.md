# Blog Backend 技术分析

## 1. 项目定位

本项目是一个个人博客网站后端服务，面向前台博客展示和后台内容管理两个场景，提供文章、项目展示、分类、标签、评论、媒体上传、站点配置、游客账号和管理员后台等 REST API。

如果用于简历，可以把它定位为：

> 基于 Spring Boot 3 + MyBatis-Plus + MySQL + Redis + Flyway 实现的个人博客内容管理后端，覆盖公开内容浏览、管理员内容管理、游客评论、图片上传、Token 登录态、热点数据缓存和 Docker Compose 部署。

项目不是单纯的 CRUD。它包含几个比较适合面试展开的技术点：

- Spring Security + Redis 不透明 Token 认证。
- 管理员与游客双角色权限模型。
- 登录失败限流和批量会话失效。
- 文章列表/详情、站点配置 Redis 缓存。
- 文章访问量异步原子递增。
- Flyway 管理 MySQL schema 演进。
- MyBatis-Plus 分页、条件查询、自动时间填充。
- 本地文件上传、静态资源映射和媒体元数据管理。
- 统一响应、统一异常、Bean Validation 参数校验。
- Dockerfile + docker-compose 一键启动 MySQL、Redis、后端应用。

## 2. 技术栈与选择原因

| 分类 | 当前选型 | 项目中的作用 | 选择原因 | 面试可讲点 |
| --- | --- | --- | --- | --- |
| 语言与运行时 | Java 17 | 后端主语言 | 与 Spring Boot 3 生态匹配，支持 record、现代 JVM 特性 | DTO/VO 使用 record 可以减少样板代码 |
| Web 框架 | Spring Boot 3.5.0 | REST API、配置装配、依赖注入 | 成熟稳定，生态完整，适合中小型业务快速落地 | 自动配置、分层设计、全局异常处理 |
| 安全框架 | Spring Security | 鉴权、认证上下文、权限规则 | 统一管理认证入口、权限失败响应和 Filter 链 | 自定义 `TokenAuthFilter` 接入 Redis 登录态 |
| ORM | MyBatis-Plus 3.5.12 | Mapper、分页、条件构造、自动填充 | 比手写 SQL 更高效，又保留 SQL 可控性 | `LambdaQueryWrapper`、`PaginationInnerInterceptor` |
| 数据库 | MySQL 8 | 持久化业务数据 | 关系模型清晰，适合文章、用户、评论、分类标签等结构化数据 | 索引设计、软删除、唯一约束 |
| 缓存 | Redis 7 | 登录态、限流、热点数据缓存 | TTL、Set、计数器都很适合本项目 | Token TTL、登录失败窗口、缓存失效策略 |
| 迁移 | Flyway | 数据库版本管理 | schema 可以随代码演进，部署时自动迁移 | `V1/V2/V3` 迁移脚本 |
| 构建 | Maven | 依赖与打包 | Spring Boot 标准构建方式 | 多阶段 Docker 构建中执行 Maven package |
| 容器化 | Docker / Compose | 本地和服务器部署 | 后端、MySQL、Redis 可一键编排 | 健康检查、volume 持久化、端口只绑定本机 |
| 参数校验 | Jakarta Validation | DTO 与 Controller 参数校验 | 将输入合法性前置，减少 Service 防御代码 | `@Valid`、`@Size`、`@Min`、`@Max` |
| 密码哈希 | BCrypt | 用户密码安全存储 | 自带盐和慢哈希，比普通摘要更适合密码存储 | 不返回明文密码或 passwordHash |

需要注意：现有 `pom.xml` 没有引入 Markdown 渲染或 HTML 清洗库，文章 `contentHtml` 当前由后端返回 `null`，主要存储和返回 `contentMarkdown`，渲染职责更偏前端。

## 3. 目录结构与分层

```text
src/main/java/com/blog
├── BlogApplication.java          # Spring Boot 启动类，启用异步与配置属性
├── common                        # 统一响应、分页、错误码、业务异常、全局异常
├── config                        # Security、MyBatis-Plus、上传静态资源映射、应用配置
├── controller                    # REST Controller，按公开接口和后台接口拆分
├── domain                        # 数据库实体与枚举
├── dto                           # 请求 DTO，集中在 Requests
├── mapper                        # MyBatis-Plus BaseMapper
├── security                      # 登录用户、认证上下文、Token 过滤器
├── service                       # 业务逻辑、事务、缓存、文件上传
├── util                          # Slug 生成、评论内容清洗
└── vo                            # 响应视图，集中在 Views
```

分层职责比较清晰：

- Controller：只负责路由、参数接收、校验触发和统一响应封装。
- Service：承载核心业务规则、事务边界、缓存读写、权限上下文读取。
- Mapper：基于 MyBatis-Plus 负责数据访问，少量自定义 SQL 用于文章浏览量原子递增。
- Domain：对应数据库表字段，枚举以字符串形式持久化。
- DTO/VO：把请求和响应模型与数据库实体隔离。
- Common：统一错误模型，降低前后端联调成本。

这种分层适合在面试中解释为：Controller 保持薄，业务规则不散落在接口层；Service 是事务与领域规则的边界；VO 防止敏感字段泄露。

## 4. 启动、配置与基础设施

### 4.1 启动类

`BlogApplication` 使用：

- `@SpringBootApplication` 启动 Spring Boot。
- `@EnableAsync` 支持异步方法，当前用于文章浏览量递增。
- `@EnableConfigurationProperties(AppProperties.class)` 绑定自定义配置。

### 4.2 应用配置

`application.yml` 使用环境变量覆盖默认值：

- `SERVER_PORT`：默认 `8080`。
- `SPRING_DATASOURCE_*`：MySQL 连接信息。
- `SPRING_DATA_REDIS_*`：Redis 连接信息。
- `APP_UPLOAD_ROOT`：上传根目录，默认 `./data/uploads`。
- `APP_TOKEN_TTL_SECONDS`：登录 token 有效期，默认 604800 秒，也就是 7 天。
- `APP_ADMIN_USERNAME` / `APP_ADMIN_PASSWORD`：启动初始化管理员账号。

这种配置方式兼顾本地开发和 Docker 部署。简历里可以说“通过环境变量实现配置外置化，避免把生产配置写死在代码中”。

### 4.3 MyBatis-Plus 配置

`MybatisPlusConfig` 做了两件事：

- 注册 `PaginationInnerInterceptor(DbType.MYSQL)`，让 `selectPage` 支持 MySQL 分页。
- 注册 `MetaObjectHandler`，自动填充 `createdAt` 和 `updatedAt`。

自动填充的好处是所有实体不用在每个 Service 中重复设置创建/更新时间，减少遗漏。

### 4.4 静态资源映射

`WebMvcConfig` 将 `/uploads/**` 映射到 `app.upload-root` 对应的本地目录。图片上传后会生成类似：

```text
/uploads/article/2026/05/18/{uuid}.jpg
```

前端可以直接通过这个 URL 访问图片。

## 5. 数据库模型设计

数据库迁移文件在 `src/main/resources/db/migration`：

- `V1__init_schema.sql`：用户、文章、分类、标签、评论、封面配置、个人信息配置、媒体资源、系统配置等基础表。
- `V2__add_project.sql`：新增项目展示表。
- `V3__add_project_content_markdown.sql`：给项目增加 Markdown 长正文。

### 5.1 核心表

| 表 | 作用 | 关键设计 |
| --- | --- | --- |
| `user` | 管理员和游客账号 | `username` 唯一，密码存 `password_hash`，角色 `ADMIN/GUEST`，状态 `ACTIVE/DISABLED` |
| `article` | 博客文章 | `slug` 唯一，Markdown 正文，发布状态，浏览量，软删除，创建/更新人 |
| `category` | 分类 | `slug` 唯一，排序字段，软删除 |
| `tag` | 标签 | `name` 与 `slug` 唯一，软删除 |
| `article_tag` | 文章标签多对多关系 | `(article_id, tag_id)` 唯一，防止重复绑定 |
| `comment` | 评论 | 绑定文章和用户，状态 `VISIBLE/HIDDEN/PENDING`，记录 IP 与 UA |
| `cover_config` | 首页封面配置 | JSON 保存链接列表，`is_active` 标记当前配置 |
| `profile_config` | 个人信息页配置 | JSON 保存社交链接，Markdown 保存长简介 |
| `media_asset` | 上传媒体资源 | 记录原始文件名、存储路径、URL、用途、上传人 |
| `system_config` | 系统配置 | key-value 配置，支持字符串、数字、布尔、JSON 类型 |
| `project` | 项目展示 | `slug` 唯一，状态、排序、标签 JSON、Markdown 长说明 |

### 5.2 软删除策略

多数业务表有 `deleted_at` 字段。删除时不是物理删除，而是设置当前时间：

- 文章删除：设置 `article.deleted_at`。
- 分类删除：先检查是否有未删除文章引用，未被引用才能软删除。
- 标签删除：设置 `tag.deleted_at`，并删除 `article_tag` 关系。
- 评论删除：设置 `comment.deleted_at`。
- 媒体删除：设置 `media_asset.deleted_at`，当前不删除物理文件。
- 项目删除：设置 `project.deleted_at`。

软删除的优点：

- 后台可扩展恢复能力。
- 审计友好。
- 避免误删造成不可逆数据丢失。

代价：

- 所有查询必须记得过滤 `deleted_at is null`。
- 唯一索引和软删除之间要谨慎处理。当前 `slug` 唯一索引会导致软删除后的相同 slug 不能复用，这是一个可优化点。

### 5.3 索引设计

项目中的索引主要围绕查询路径设计：

- `article(status, published_at)`：前台文章列表按发布状态和发布时间查询。
- `article(category_id)`：按分类筛选文章。
- `article(created_by)`：按创建人追溯。
- `comment(article_id, status, created_at)`：文章详情页加载可见评论。
- `comment(user_id)`：后台按用户追溯评论。
- `project(status, sort_order, id)`：前台项目列表按状态和排序展示。
- `media_asset(usage_type)`、`media_asset(uploaded_by)`、`media_asset(created_at)`：后台按用途、上传人、时间管理资源。
- `user(username)`：登录和注册唯一性校验。

面试可以强调：索引不是按字段随便加，而是围绕高频查询和排序条件设计。

## 6. API 与统一响应

所有接口返回 `ApiResponse<T>`：

```json
{
  "code": "SUCCESS",
  "message": "success",
  "data": {}
}
```

分页接口返回 `PageResponse<T>`：

```json
{
  "items": [],
  "page": 1,
  "size": 10,
  "total": 100,
  "pages": 10
}
```

统一响应的价值：

- 前端可以用一套拦截器处理成功、失败、提示和登录失效。
- 后端错误码由 `ErrorCode` 统一管理。
- 分页格式统一，减少不同接口的适配成本。

主要接口分为三类：

- 公开接口：`/api/cover`、`/api/profile`、`/api/articles`、`/api/projects`、`/api/categories`、`/api/tags` 等。
- 认证接口：`/api/auth/login`、`/api/auth/logout`、`/api/auth/me`、`/api/auth/guest/register`。
- 后台接口：`/api/admin/**`，只有管理员可访问。

## 7. 认证与授权实现

### 7.1 为什么使用 Redis Token 而不是 JWT

项目采用“不透明随机 token + Redis 存储登录态”的方案：

1. 登录成功后生成两个 UUID 拼接的随机字符串。
2. Redis 写入 `auth:token:{token}`，value 是序列化后的 `LoginUser`。
3. 设置 TTL，默认 7 天。
4. 请求携带 `Authorization: Bearer <token>`。
5. `TokenAuthFilter` 根据 token 查询 Redis，存在则写入 Spring Security 上下文。

相比 JWT，这种方案的优势是：

- 服务端可主动失效 token，适合退出登录、改密码、禁用用户。
- token 内容不暴露给客户端。
- 权限变化可以通过删除 Redis 登录态立即生效。

代价是：

- 每次认证都需要访问 Redis。
- Redis 不可用会影响登录态校验。
- 多服务部署时 Redis 是认证链路上的关键依赖。

### 7.2 登录流程

`AuthService.login` 的核心流程：

1. 根据用户名和客户端 IP 构造限流 key：`rate:login:{username}:{ip}`。
2. 如果 15 分钟窗口内失败次数达到 5 次，返回 `RATE_LIMITED`。
3. 查询用户，校验用户存在、状态为 `ACTIVE`、BCrypt 密码匹配。
4. 登录失败时 Redis 计数加一，并设置 15 分钟过期。
5. 登录成功后删除失败计数，更新 `lastLoginAt`。
6. 生成 token，写入 `auth:token:{token}`。
7. 同时维护 `auth:user:{userId}:tokens` Set，便于批量踢下线。

### 7.3 批量会话失效

`AuthService.invalidateUserTokens(userId)` 会读取：

```text
auth:user:{userId}:tokens
```

然后逐个删除：

```text
auth:token:{token}
```

触发场景：

- 管理员重置游客密码。
- 管理员禁用游客。
- 管理员修改自己的密码。

这是简历上比较好讲的点：不是简单登录，而是考虑到了账号安全状态变化后的会话治理。

### 7.4 权限控制

`SecurityConfig` 中的规则：

- `POST /api/auth/guest/register`、`POST /api/auth/login`：公开。
- 文章、项目、分类、标签、封面、信息页公开 GET 接口：公开。
- `/uploads/**`：公开访问。
- `/api/admin/**`：需要 `ADMIN`。
- `/api/auth/logout`、`/api/auth/me`：需要 `GUEST` 或 `ADMIN`。
- `POST /api/articles/*/comments`：需要 `GUEST` 或 `ADMIN`。

`TokenAuthFilter` 读取登录态后创建 `UsernamePasswordAuthenticationToken`，权限格式是：

```text
ROLE_ADMIN
ROLE_GUEST
```

所以 Spring Security 的 `hasRole("ADMIN")` 能够正常匹配。

## 8. 核心业务模块

### 8.1 文章模块

文章模块由 `ArticleService` 承担核心逻辑。

前台列表：

- 只查询 `status = PUBLISHED` 且未软删除文章。
- 支持 `categorySlug`、`tagSlug`、`keyword` 过滤。
- 按 `publishedAt` 和 `id` 倒序。
- 使用 Redis 缓存 5 分钟，key 形如：

```text
article:list:{page}:{size}:{filterHash}
```

前台详情：

- 根据 `slug` 查询已发布文章。
- 详情缓存 10 分钟，key 形如：

```text
article:detail:{slug}
```

- 命中缓存后仍会异步增加浏览量。

后台管理：

- 可按状态、关键词、分类分页查询。
- 创建文章时 slug 为空则根据标题生成。
- 创建发布态文章时设置 `publishedAt`。
- 草稿或隐藏文章首次改为发布时设置 `publishedAt`，之后再编辑不覆盖首次发布时间。
- 创建/更新文章与标签关系在同一个事务中完成。
- 删除文章使用软删除。

标签关系维护：

- 更新文章时先删除旧的 `article_tag` 关系。
- 对新的 `tagIds` 去重后重新插入。
- `tagService.ensureAllExist` 确保所有标签都存在且未删除。

缓存失效：

- 文章创建、更新、删除后删除详情缓存。
- 使用 `SetUtils.deleteByPattern(redisTemplate, "article:list:*")` 删除所有文章列表缓存。

可讲的取舍：

- 列表缓存按分页和筛选参数区分，避免不同条件互相污染。
- 详情缓存和异步浏览量分离，保证读性能。
- 当前列表缓存清理使用 Redis `keys`，小项目足够；生产大规模 key 建议改为 `SCAN` 或缓存版本号。

### 8.2 分类与标签模块

分类 `CategoryService`：

- 前台按 `sortOrder`、`id` 升序展示。
- 后台支持创建、更新、删除。
- slug 自动生成并校验唯一。
- 删除前检查是否被未删除文章引用，被引用则返回 `CONFLICT`。

标签 `TagService`：

- 前台按名称升序展示。
- 创建/更新时校验 name 或 slug 不重复。
- 删除标签时软删除标签，并删除文章标签关系。
- 提供 `viewsByArticleId` 给文章 VO 组装标签列表。

设计原因：

- 分类是一对多关系，删除时如果存在文章引用，直接删除会破坏文章分类展示，所以选择阻止删除。
- 标签是多对多关系，删除标签时同步清理关系表，避免脏关系。

### 8.3 评论模块

评论由 `CommentService` 实现。

公开评论列表：

- 先通过 slug 找到已发布文章。
- 只返回 `status = VISIBLE` 且未软删除的评论。
- 按创建时间倒序分页。

发表评论：

- 需要登录用户，游客和管理员都可以。
- 评论必须绑定已发布文章。
- 评论内容经过 `ContentSanitizer.cleanComment` 处理，本质是 trim 后 HTML escape。
- 记录客户端 IP 和 User-Agent，用于后台审计。
- 默认状态来自系统配置 `comment.defaultStatus`，默认 `VISIBLE`，也可以改成 `PENDING` 做审核流。

后台管理：

- 支持按状态、文章 ID、用户名过滤。
- 可修改评论状态。
- 删除使用软删除。

可讲的安全点：

- 评论响应中只返回作者 ID、用户名、昵称，不返回密码哈希。
- 用户输入做 HTML escape，降低 XSS 风险。
- UA 做长度截断，避免超长 header 写库。

可优化点：

- 当前 `toCommentView` 会按评论逐条查用户，后台列表还会查文章，存在 N+1 查询风险。
- 可通过批量查询用户和文章后在内存 Map 组装，或写联表查询 VO。

### 8.4 用户与账号管理

用户分两类：

- `ADMIN`：后台管理员。
- `GUEST`：游客用户，主要用于登录后评论。

游客注册：

- 用户名 3-64 位，只允许字母、数字、下划线、短横线。
- 密码 6-64 位。
- 昵称可选，不填时使用用户名。
- 密码使用 BCrypt 哈希写入 `password_hash`。

管理员初始化：

- `StartupInitializer` 在应用启动时检查配置中的管理员用户名是否存在。
- 不存在则创建管理员账号。
- 默认账号密码来自 `APP_ADMIN_USERNAME` 和 `APP_ADMIN_PASSWORD`。

后台用户管理：

- 分页查询游客。
- 重置游客密码。
- 启用或禁用游客。
- 修改管理员自己的密码。

安全取舍：

- 管理员只能重置游客密码，不能查看游客原密码。
- 密码修改和用户禁用会让 Redis 中已有 token 失效。
- 当前默认管理员密码在配置中有默认值，生产环境必须覆盖。

### 8.5 站点配置模块

`SiteService` 管理封面、个人信息页和系统配置。

封面配置：

- 从 `cover_config` 查询 `is_active = true` 的记录。
- `links_json` 反序列化为 `List<LinkItem>`。
- 缓存 key：`site:cover`，TTL 30 分钟。
- 后台更新后删除缓存。

个人信息页配置：

- 从 `profile_config` 查询 `is_active = true` 的记录。
- `social_links_json` 反序列化为结构化列表。
- 支持 `contentMarkdown` 长内容。
- 缓存 key：`site:profile`，TTL 30 分钟。

系统配置：

- `system_config` 存储评论默认状态、上传大小、允许图片类型等配置。
- 后台可更新配置值、类型和描述。

可讲的设计原因：

- 把封面和个人信息页做成配置表，避免每次改文案或链接都重新部署。
- JSON 字段适合链接列表这种结构简单、查询频率低、主要整体读写的配置数据。

### 8.6 媒体上传模块

`MediaService.upload` 实现图片上传。

校验规则：

- 文件不能为空。
- MIME 类型必须是 `image/jpeg`、`image/png`、`image/webp`、`image/gif` 之一。
- MIME 类型还要出现在系统配置 `upload.allowedImageTypes` 中。
- 文件大小不能超过系统配置 `upload.maxFileSizeMb`。
- 文件扩展名需要和 MIME 类型匹配。
- 原始文件名通过 `Path.of(original).getFileName()` 和字符替换做基础清理。

存储规则：

```text
{uploadRoot}/{usageType}/{yyyy}/{MM}/{dd}/{uuid}.{ext}
```

URL 规则：

```text
/uploads/{usageType}/{yyyy}/{MM}/{dd}/{uuid}.{ext}
```

元数据写入 `media_asset`：

- 原始文件名。
- 存储文件名。
- MIME 类型。
- 文件大小。
- 存储类型，当前是 `LOCAL`。
- 物理路径。
- 可访问 URL。
- 使用场景 `ARTICLE/COVER/PROFILE/PROJECT/OTHER`。
- 上传人。

一致性处理：

- 先写文件，再写数据库。
- 如果数据库写入失败，会删除已经写入的物理文件作为补偿。

当前不足：

- 删除媒体时只软删除数据库记录，不删除物理文件。
- 只基于 `MultipartFile.getContentType()` 和扩展名校验，没有读取文件魔数。
- 本地存储适合单机和小项目，多实例部署需要共享存储或对象存储。

### 8.7 项目展示模块

项目模块由 `ProjectService` 实现，适合展示个人作品或实践项目。

功能：

- 公开列表：只展示 `PUBLISHED` 且未软删除项目。
- 公开详情：通过 slug 查询项目。
- 后台列表：支持状态和关键词筛选。
- 后台创建、更新、删除。
- 支持 `contentMarkdown` 长说明。
- 支持 `tags`，内部以 JSON 存储在 `tags_json`。
- 返回只读字段 `detailUrl = /projects/{slug}`，方便前端卡片跳转详情页。

项目标签为什么用 JSON 而不是单独建表：

- 项目标签更像展示用的轻量元信息，当前没有复杂筛选或统计需求。
- JSON 简化了模型和接口。
- 如果后续需要按项目标签聚合、筛选、推荐，可以再规范化成 `project_tag` 表。

## 9. Redis 使用设计

| 用途 | Key | TTL | 说明 |
| --- | --- | --- | --- |
| 登录态 | `auth:token:{token}` | 7 天 | 保存序列化 `LoginUser` |
| 用户 token 索引 | `auth:user:{userId}:tokens` | 7 天 | 支持批量会话失效 |
| 登录失败限流 | `rate:login:{username}:{ip}` | 15 分钟 | 失败 5 次后限制登录 |
| 文章列表缓存 | `article:list:{page}:{size}:{filterHash}` | 5 分钟 | 缓存公开文章列表 |
| 文章详情缓存 | `article:detail:{slug}` | 10 分钟 | 缓存公开文章详情 |
| 封面缓存 | `site:cover` | 30 分钟 | 缓存首页封面 |
| 个人信息缓存 | `site:profile` | 30 分钟 | 缓存个人信息页 |

缓存策略：

- 读多写少的数据使用缓存。
- 写操作完成后删除相关缓存，采用 cache-aside 模式。
- JSON 反序列化失败时删除损坏缓存，回源数据库。

面试可讲：

- 为什么文章详情命中缓存还要增加浏览量：浏览量是行为统计，不应该因为缓存命中而丢失。
- 为什么文章列表缓存的 key 包含 filterHash：分类、标签、关键词组合较多，用 hash 可以避免 key 过长。
- 为什么使用 TTL：即使缓存失效漏删，也能最终回源数据库。

## 10. 事务边界

项目中的关键事务：

| 场景 | 事务方法 | 原因 |
| --- | --- | --- |
| 游客注册 | `AuthService.registerGuest` | 用户创建和密码哈希保存必须一致 |
| 登录 | `AuthService.login` | 更新 lastLoginAt 与生成登录态在一个业务流程中 |
| 创建/更新文章 | `ArticleService.create/update` | 文章主表与标签关系需要一致 |
| 删除文章 | `ArticleService.delete` | 软删除与审计字段更新一致 |
| 创建/更新/删除分类 | `CategoryService` | 唯一性校验、引用检查和写入一致 |
| 创建/更新/删除标签 | `TagService` | 标签与关系表清理一致 |
| 创建评论 | `CommentService.create` | 评论内容、作者、审计信息一起写入 |
| 修改评论状态/删除评论 | `CommentService` | 状态变更或软删除原子完成 |
| 上传文件 | `MediaService.upload` | 数据库写入失败时做文件补偿 |
| 站点配置更新 | `SiteService.updateCover/updateProfile` | DB 更新成功后失效缓存 |
| 项目创建/更新/删除 | `ProjectService` | slug 校验与项目写入一致 |

注意：文件系统和数据库不是同一个事务资源，当前通过 try/catch 做补偿删除。这是工程上常见的“最终一致 + 补偿”处理。

## 11. 安全设计

已实现的安全措施：

- 密码使用 BCrypt，不保存明文。
- 响应 VO 不包含 `passwordHash`。
- 管理接口统一走 `/api/admin/**` 并要求 `ADMIN` 角色。
- 游客只能访问发表评论、退出登录、查看当前用户等有限接口。
- 登录失败按用户名和 IP 做 Redis 窗口限流。
- 修改密码或禁用用户后主动清理 Redis token。
- 评论内容做 HTML escape。
- 上传限制 MIME、扩展名、大小和使用场景。
- 全局异常对未知错误只返回 `internal error`，不暴露堆栈。

需要补强的安全点：

- `X-Forwarded-For` 当前直接取第一个 IP，需要结合可信反向代理配置，否则可能被伪造。
- 上传校验未读取文件头魔数，攻击者可能伪造 Content-Type。
- 后台接口没有细粒度操作审计日志。
- 生产环境必须修改默认管理员密码和数据库密码。
- 当前没有 CSRF，因为采用无状态 Bearer Token；如果未来使用 Cookie 存 token，需要重新评估 CSRF。
- 评论 HTML escape 属于基础防护，如果未来后端渲染 Markdown/HTML，需要引入白名单清洗。

## 12. 部署设计

### 12.1 Dockerfile

Dockerfile 使用多阶段构建：

1. `maven:3.9-eclipse-temurin-17` 阶段复制 `pom.xml` 和 `src`，执行 `mvn -B -DskipTests package`。
2. `eclipse-temurin:17-jre` 阶段只复制打包后的 jar。
3. 设置 `APP_UPLOAD_ROOT=/app/uploads`。
4. 暴露 8080，使用 `java -jar /app/app.jar` 启动。

优势：

- 运行镜像不包含 Maven 和源码，体积更小。
- 构建环境和运行环境分离。

### 12.2 docker-compose

Compose 包含三个服务：

- `mysql`：MySQL 8.4，挂载 `mysql-data`。
- `redis`：Redis 7.4，挂载 `redis-data`。
- `app`：后端服务，依赖 MySQL 和 Redis 健康检查。

端口绑定策略：

- MySQL：`127.0.0.1:3306:3306`。
- Redis：`127.0.0.1:6379:6379`。
- App：`127.0.0.1:18080:8080`。

这样避免数据库和 Redis 直接暴露到公网。生产环境可由宿主机 Nginx 代理 `/api/` 和 `/uploads/` 到 `127.0.0.1:18080`。

## 14. 当前实现亮点

### 14.1 统一接口协议

项目用 `ApiResponse` 和 `PageResponse` 固定响应格式，让前端处理成功、失败、分页时不需要为每个接口写特例。

### 14.2 Redis 登录态可控

选择 Redis token 而不是 JWT，使退出登录、重置密码、禁用用户后立即踢下线成为可能。

### 14.3 内容缓存与写后失效

公开文章列表、文章详情、封面、个人信息页都属于读多写少数据，缓存能降低数据库压力。写操作后删除缓存，保证用户看到较新的内容。

### 14.4 浏览量异步递增

文章详情接口返回时不等待浏览量更新，减少接口响应耗时。数据库层使用：

```sql
update article set view_count = view_count + 1 where id = #{id}
```

这是原子递增，避免并发请求下读改写造成丢失更新。

### 14.5 Flyway 管理 schema 演进

项目管理模块通过 `V2` 新增表，再通过 `V3` 增加字段，体现了数据库结构随功能演进的工程化管理。

### 14.6 上传文件补偿处理

文件先落盘，再写媒体表；数据库失败时删除已写文件。虽然不是分布式事务，但对本地文件场景是实用的补偿设计。

## 15. 已知不足与优化方向

### 15.1 缓存清理方式优化

当前文章列表缓存通过 `redisTemplate.keys("article:list:*")` 清理。小项目可接受，但生产上 `KEYS` 可能阻塞 Redis。

优化方案：

- 使用 `SCAN` 分批删除。
- 使用缓存版本号：`article:list:v{version}:...`，文章变更时递增版本。
- 用 Caffeine + Redis 组合做二级缓存，但复杂度会上升。

### 15.2 查询 N+1 优化

文章列表组装时，每篇文章会查分类和标签；评论列表组装时，每条评论会查作者，后台评论还会查文章。

优化方案：

- 批量收集 ID，一次查询分类、标签、用户、文章后 Map 组装。
- 对复杂列表写 MyBatis XML 联表查询。
- 对文章摘要做冗余字段或读模型，但要处理一致性。

### 15.3 上传安全增强

当前上传基于 MIME 和扩展名判断。

优化方案：

- 读取文件头魔数验证真实类型。
- 对图片进行重新编码，去除潜在恶意 payload 和 EXIF。
- 接入对象存储，如 OSS、S3、MinIO。
- 生成缩略图和 WebP，提高前台加载速度。

### 15.4 评论风控

当前只有登录后评论和基础 HTML escape。

优化方案：

- 评论默认进入 `PENDING` 审核。
- 增加发表评论频率限制。
- 增加敏感词过滤。
- 同一文章同一用户短时间重复内容拦截。

### 15.5 认证体系增强

当前是单 token，有 TTL。

优化方案：

- 引入 refresh token / access token 双 token。
- 记录设备信息、登录 IP、登录地点。
- 后台提供会话列表和主动踢下线。
- 管理员账号启用二次验证。

### 15.6 数据库约束增强

当前 schema 主要靠索引和业务代码约束，没有显式外键。

优化方案：

- 如果系统规模小且强一致优先，可以加外键约束。
- 如果希望业务灵活和迁移方便，保留无外键方案，但要补充 Service 层引用检查和数据巡检。
- 对软删除唯一键可改成组合唯一，如 `(slug, deleted_at)` 在 MySQL 下需注意 NULL 唯一语义，或引入 `is_deleted`。

### 15.7 可观测性增强

当前没有日志、指标、链路追踪体系。

优化方案：

- 关键操作记录审计日志：登录、改密码、发文、删文、上传、评论审核。
- 接入 Spring Boot Actuator。
- 对接口耗时、错误率、缓存命中率做指标。
- 统一请求 ID，便于排查线上问题。

### 15.8 测试增强

优化方案：

- 使用 Testcontainers 启动 MySQL 和 Redis 做集成测试。
- 使用 MockMvc 测试 Security 权限规则。
- 针对 Flyway migration 做启动测试。
- 针对上传失败补偿做边界测试。
- 针对缓存命中、缓存失效做 Service 测试。

## 16. 面试官可能追问的问题

### Q1：为什么不用 JWT，而用 Redis Token？

因为项目需要主动控制登录态。例如退出登录、管理员重置游客密码、禁用游客、管理员修改密码后，都希望旧 token 立刻失效。JWT 天然无状态，签发后在过期前通常无法主动撤销，除非再加黑名单；既然还是要查 Redis，不如直接使用不透明 token + Redis 存储登录态，逻辑更直接。

### Q2：Redis 挂了怎么办？

当前实现中 Redis 是认证和缓存依赖。Redis 不可用会影响登录态校验和限流，缓存不可用也会导致接口异常。生产优化可以：

- Redis 部署主从或哨兵。
- 认证链路对 Redis 做超时控制。
- 缓存读取失败时可降级回源数据库，但登录态不能随意放行。
- 对管理接口保持严格失败关闭，对公开内容接口可做降级。

### Q3：浏览量为什么异步更新？

文章详情是高频读接口，浏览量不是影响响应内容正确性的强一致字段。异步递增可以缩短详情接口响应时间。递增 SQL 使用数据库原子更新，避免并发下丢失更新。

### Q4：缓存和数据库如何保证一致？

采用 cache-aside 模式。读时先查缓存，未命中查数据库并写缓存；写操作成功后删除相关缓存。再加 TTL 作为兜底，即使某次删除失败，缓存也会最终过期。这个项目是博客场景，对内容更新后的瞬时强一致要求不高，所以这种方案性价比高。

### Q5：文章 slug 如何生成和防冲突？

`SlugUtil` 会把标题标准化、小写化，去掉音调符号，把非字母数字中文的字符替换为短横线，并去掉首尾短横线。创建和更新时通过数据库查询保证未删除文章中 slug 唯一。如果冲突返回 `CONFLICT`。

### Q6：文章、分类、标签之间怎么设计？

文章和分类是一对多，文章表保存 `category_id`。文章和标签是多对多，用 `article_tag` 关系表，并对 `(article_id, tag_id)` 加唯一索引。分类删除前检查是否有文章引用；标签删除时清理关系表。

### Q7：为什么项目标签用 JSON，文章标签用关系表？

文章标签用于文章筛选、列表展示和分类检索，属于核心查询维度，所以用关系表。项目标签当前主要用于展示，不承担复杂筛选和统计，所以用 JSON 简化设计。后续如果项目标签也要支持筛选和聚合，可以迁移成关系表。

### Q8：如何避免密码泄露？

请求里接收密码后立即用 BCrypt 编码，数据库只保存 `password_hash`。响应模型使用 `UserView`、`GuestView` 等 VO，不包含密码字段。管理员也只能重置游客密码，不能查看原密码。

### Q9：上传文件如何防止路径穿越？

存储文件名由服务端生成 UUID，不使用用户传入文件名作为存储路径。原始文件名只作为展示元数据，并通过 `Path.of(original).getFileName()` 提取文件名，再替换斜杠、换行等危险字符。

### Q10：为什么没有使用数据库外键？

当前项目主要通过 Service 层做引用检查，例如删除分类前检查文章引用，删除标签时清理关系表。无外键的好处是部署和迁移更灵活，坏处是数据一致性更多依赖业务代码。生产系统可以根据一致性要求决定是否加外键或增加数据巡检任务。

### Q11：全局异常怎么设计？

业务异常统一抛 `BizException(ErrorCode, message)`，由 `GlobalExceptionHandler` 转成统一响应。参数校验错误、未授权、无权限、上传过大和未知异常也统一处理。未知异常只返回 `internal error`，避免暴露内部堆栈。

### Q12：如何保证文章更新和标签关系更新一致？

创建和更新文章方法使用 `@Transactional`。文章主表写入和 `article_tag` 删除/插入在同一个事务里，如果任何一步失败，事务回滚，避免文章已更新但标签关系半更新。

### Q13：系统配置为什么放数据库？

评论默认状态、上传大小、允许图片类型这类配置希望后台可改，不希望每次调整都重新发版。放 `system_config` 后可以通过后台接口修改，Service 读取配置并应用。

### Q14：这个项目最大的技术风险是什么？

当前最大风险不是业务不可用，而是生产级能力还不完整：缓存清理用了 `KEYS`，列表组装有 N+1 查询，上传文件安全校验还可以增强，集成测试覆盖不足。这些都可以作为后续优化路线。

- 使用 Flyway 管理 MySQL 表结构演进，结合 Docker Compose 编排 Spring Boot、MySQL、Redis，实现本地和服务器环境一键部署。

如果简历空间更短，可以压缩成：

> 个人博客后端：Spring Boot 3 + MyBatis-Plus + MySQL + Redis + Flyway。负责文章/项目/评论/媒体/后台管理 REST API，实现 Redis Token 登录态、角色鉴权、登录限流、热点内容缓存、异步浏览量统计、文件上传和 Docker Compose 部署。

## 18. 项目答辩思路

面试时可以按这个顺序讲：

1. 先讲项目解决什么问题：个人博客前台展示 + 后台内容管理。
2. 再讲架构：Spring Boot 分层，Controller/Service/Mapper/DTO/VO。
3. 重点讲认证：Redis Token、Spring Security Filter、角色权限、会话失效。
4. 讲文章模块：发布状态、slug、分类标签、缓存、异步浏览量。
5. 讲工程化：Flyway、Docker Compose、统一异常、参数校验。
6. 最后主动讲不足和优化：N+1、Redis KEYS、上传安全、集成测试。

这种讲法的好处是：既展示了已经完成的功能，也能体现你知道当前系统离生产级还有哪些差距，以及下一步怎么演进。
